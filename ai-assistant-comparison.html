两个 AI 助手 对比

tools-briefing 项目内共存的两个聊天组件，技术栈和能力完全不同

🤖

简报区 LLM 助手

chat.js + Cloudflare Worker
部署页面7 页（首页+6期简报）
后端DeepSeek LLM
多轮对话✓ 支持
知识范围系统提示词（固定知识库）
UI 位置右下角浮动按钮
代码量217 行（chat.js + worker.js）
维护独立 JS 文件，一改全站生效
费用DeepSeek API 按量（有成本）
🔍

研究报告搜索助手

reports/index.html 内联
部署页面1 页（reports/index.html）
后端无（纯前端）
多轮对话✗ 不支持
知识范围28 份报告标题+关键词数组
UI 位置页面底部嵌入块
代码量~70 行（内联在 HTML 里）
维护数据写死在 HTML，更新需改代码
费用零成本

🔧 架构对比

① 简报区 LLM 助手 — 请求链路

🖥️用户输入 → chat.js
↓
☁️Cloudflare Worker 代理
↓
🧠DeepSeek API /chat/completions
↓
💬流式返回 → 渲染气泡

② 研究报告搜索助手 — 请求链路

🖥️用户输入 → askAI()
↓
📋reportIndex 数组关键词匹配
↓
📊返回匹配报告列表（≤5条）
↓
💬渲染结果 → 点击跳转定位

📏 代码体量

LLM 助手 chat.js
128 行
LLM Worker
89 行
搜索助手内联 JS
~70 行
💡

关键差异：一个有脑子，一个只是"搜一下"

① 简报区助手是真正的 LLM 对话机器人——能理解上下文、自由回答、跨话题聊天；② 研究报告助手只是把 28 条数据写死在 JS 数组里做关键词匹配，本质是带聊天气泡外皮的搜索框。两者的代码互不共享，如果要改 UI 或换后端，要改两处。
