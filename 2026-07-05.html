01

MCP 2026-07-28 RC——Agent 基础设施的"无状态"拐点

5 月 21 日，MCP 团队锁定 2026-07-28 Release Candidate——这是 MCP 自 2024 年 11 月发布以来最大的一次协议修订。6 大 Breaking Changes 同时抵达，7 月 28 日最终规范发布，所有生产 MCP 服务器需要在此之前完成迁移。

六大 Breaking Changes

1. Session 移除协议层不再有 session 概念客户端元数据、能力、协议版本现在通过每个请求的 _meta 字段携带。不再需要 session handshake 和 session store。
2. 新 HTTP 请求头Mcp-Method + Mcp-NameStreamable HTTP 传输现在要求这两个头。服务器必须在请求头和 body 不一致时拒绝。
3. Extensions 框架MCP Apps + Tasks官方扩展框架使得第三方可以实现 MCP Apps 和 Tasks，而不需要修改核心协议。
4. Tasks 扩展重新设计的任务系统支持长时间运行的任务、进度报告、取消操作——agent 不只调 tool，还能跟踪任务状态。
5. 授权加固更强的权限边界OAuth 2.1 + PKCE 成为推荐授权方式。授权流程与协议核心解耦。
6. 正式弃用策略版本化弃用MCP 现在有正式的弃用政策——旧功能不会突然消失，但有明确的退役时间线。

为什么是拐点：MCP 正在从"实验性协议"变成"无聊但可靠的基础设施层"。之前 MCP 部署需要 sticky session 路由 + 共享 session store + 复杂的网关配置，无状态化后只需要 round-robin 负载均衡——部署复杂度骤降。结合 78% 企业 AI 团队已在生产使用 MCP、9400+ 公开服务器、9700 万月 SDK 下载的数据来看，MCP 已经跨过了从"早期采用"到"主流部署"的鸿沟。

7 月 28 日的意义：这不是"MCP 2.0"——官方称之为"最大修订"而不是版本号跳跃。但实际影响比版本号变化更大：无状态化将根本性改变 MCP 的部署模式、运维模型、以及生态系统的发展方向。对 MCP 服务器维护者来说，接下来的 3-4 周是关键的迁移窗口期。

判标说明：🔥 如果你维护 MCP 服务器，现在就需要读 RC 并制定迁移计划。无状态化的好处是真实且巨大的——部署简化意味着更多组织能低门槛接入 MCP。如果你在评估 MCP 生态，这次变革消除了 MCP 部署复杂度的最大痛点。
02

Agent 基础设施品类爆发——网关、测试、协作、浏览器四层同时在建

本周 GitHub Trending 和 Product Hunt 集中出现了多个 Agent 基础设施品类。这些不是传统工具的"agent 版包装"，而是专门为 AI agent 设计的原生基础设施——它们在解决 agent 特有的问题：模型路由、沙盒测试、多对多协作、浏览器交互。

Agent 基础设施四层架构

网关层OmniRoute231+ 模型提供商单端点路由，含 token 压缩 + 自动 fallback，+387 stars/day
测试/沙盒层TryCaseAI Coding Agent 一次性测试环境——给 agent 一个沙盒跑代码验证后再提交。PH 170+ votes
Termi Protocol实时 3D 可视化 AI coding agent 的构建过程。PH 170+ votes
协作层CircleChat给 AI agent 一个 Slack + 任务板 + manager——单 agent 到多 agent 协作
N71跨 agent 共享上下文，防止信息孤岛
浏览器层Safari MCP ServerWebKit 官方出品，agent 驱动 Safari 浏览器。HN 269 points
BrowserActagent 专用浏览器自动化——过验证页、填表单、上传文件、登录态操作

为什么是现在：当 agent 从"偶尔用一下"变成"基础设施上运行的持续性工作负载"时，agent 网关、agent 测试、agent 协作就变成了独立的品类机会。TryCase 解决的是真实痛点——给 AI 代码一个沙盒测试，验证通过后再合并，这是 agent 代码安全的基础设施。CircleChat 解决的是另一个真实痛点——多 agent 协作时，人类需要一个观察和管理层。

从单 agent 到多 agent 基础设施：

  • 网关层（OmniRoute）：路由到最优模型——"不管用哪个 agent，都自动选最好的模型和 provider"
  • 沙盒层（TryCase/Termi Protocol）：安全测试——"agent 写的代码先跑在沙盒里，通过再合并"
  • 协作层（CircleChat/N71）：多 agent 管理——"一个 Slack 频道里管理多个 agents 的对话和进度"
  • 浏览器层（Safari MCP/BrowserAct）：原生浏览器交互——"agent 不再用 hacky 的方式驱动浏览器"
判标说明：🔥 四层架构的出现标志着 Agent 从"工具"走向"平台"。下半年的竞争不是"谁做了更好的 agent"，而是"谁建了更好的 agent 基础设施"。如果你在做 agent 产品或重度使用 agent，这四层都需要纳入基础设施评估。
补充判标：⚡ Product Hunt 7 月 2-5 日集中了 TryCase、CircleChat、MentionDrop MCP、Termi Protocol、DocsAlot、BrowserAct 等多个 agent 基础设施工具——这不是巧合，是品类正在形成的信号。但大部分工具还很早期（<200 votes），成熟度需要持续跟踪。

📊 趋势洞察

趋势 1

MCP 进入"基础设施化"阶段——无状态协议改写部署模式

MCP 2026-07-28 RC 是 MCP 从"协议"到"基础设施"的关键转折点。无状态化意味着 MCP 服务器可以像普通 HTTP 服务一样部署——不需要 sticky session、不需要共享 session store、不需要复杂的网关配置。之前需要 Redis session store + L7 负载均衡的 MCP 部署，现在可以在 round-robin 负载均衡后面直接跑。结合 78% 企业 AI 团队已在生产使用 MCP 的数据，MCP 正在从"实验性协议"变成"无聊但可靠的基础设施层"——就像 HTTP 和 TCP 一样。

传导链：无状态协议 → 部署简化 → 企业采纳加速 → MCP 网关/安全/治理品类爆发 → MCP 从可选变成默认。接下来的 3-4 周（7 月 28 日前）是 MCP 服务器大规模迁移的窗口期。

趋势 2

Agent 基础设施品类从零到一——网关、测试、协作、浏览器四层同时在建

本周 GitHub Trending + Product Hunt 集中出现了多个 Agent 基础设施品类：OmniRoute（网关层）、TryCase + Termi Protocol（测试/沙盒层）、CircleChat + N71（协作层）、Safari MCP Server + BrowserAct（浏览器层）。这不是巧合——当 agent 从"偶尔用一下"变成"基础设施上运行的持续性工作负载"时，agent 网关、agent 测试、agent 协作都变成了独立的品类机会。TryCase 解决"agent 写的代码先验证再合并"，CircleChat 解决"多 agent 协作需要管理面"，BrowserAct 解决"agent 需要一个真正的浏览器而不是模拟"——每个都在解决一个 agent 特有的基础设施问题。

传导链：Agent 使用频率↑ → 需要专门的网关/测试/协作/浏览器层 → Agent 基础设施品类从零到一。下半年的竞争不是"谁做了更好的 agent"，而是"谁建了更好的 agent 基础设施"。

给 MCP 服务器维护者：读 2026-07-28 RC 并开始迁移规划

# 从 modelcontextprotocol.io 获取最新 RC spec
# 重点验证：

WebKit 官方发布的 MCP 服务器，agent 驱动浏览器有了平台级方案。性能和兼容性都是原生级别，不需要第三方工具。
