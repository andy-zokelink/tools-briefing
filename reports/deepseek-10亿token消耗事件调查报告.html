二、根因

主因 Claude Code 与 Hermes 共用同一把 DeepSeek API Key，CC 在后台大量使用 deepseek-v4-flash 执行子代理任务，累计消耗 10 亿 token。

消耗链路：

# Claude Code 配置（~/.claude.json） ANTHROPIC_BASE_URL: "https://api.deepseek.com/anthropic" ANTHROPIC_DEFAULT_HAIKU_MODEL: "deepseek-v4-flash" ANTHROPIC_SMALL_FAST_MODEL: "deepseek-v4-flash" ANTHROPIC_AUTH_TOKEN: "sk-e7b...7c9c" ← 阿森的key

CC 每处理一个 MKS 任务，会在后台启动大量子代理：

  • 读文件 → flash（每篇笔记一次）
  • 分析内容 → flash（子代理并行处理）
  • 生成 HTML → pro（主代理汇总）

一个 300 篇的课程，flash 轻松消耗数千万 token。264 个 CC 会话跑了一个月，累计 10 亿。

四、为何没及时发现

  • CC 没有 token 预算限制 —— 只关心"完成任务"，不关心花多少钱
  • CC 子代理没有超时熔断 —— 死循环了没人知道，一个进程挂了 36 天无人察觉
  • CC 和 Hermes 没有隔离 —— 共用一把 key，各跑各的，互不可见
  • 没有消费监控 —— DeepSeek 没有消费预警，月底看账单才发现

六、后续建议

  1. CC 重度任务走独立 key，设置月消费上限
  2. 评估是否将 CC 的 flash 模型换为更便宜的方案
  3. 长期考虑在 Hermes 侧加一个"全局 token 预算"的概念
